Overview of Lua’s Garbage Collector

Lua provides automatic memory management. The garbage collector (GC) traces objects that are reachable from a root set (global variables, local variables in call frames, registry, etc.) and frees those that are not reachable
lua.org
. An object is considered dead only when the collector can be certain that it will not be accessed again in the program’s normal execution
lua.org
. Lua manages memory for strings, tables, functions, threads, userdata and internal structures
lua.org
. Memory allocated through the C API remains alive as long as there is a C reference to it via the registry
lua.org
.
Mark‑and‑Sweep Foundation

Early Lua versions (≤ 5.0) used a stop‑the‑world mark‑and‑sweep collector: the collector marks reachable objects from the root set and then sweeps unmarked objects to reclaim memory
lua.org
. This approach pauses the program for the duration of a full collection, which can be noticeable in real‑time applications
lua.org
. Since Lua 5.1, the GC has been incremental and generational
lua.org
.
Incremental GC

Incremental mode interleaves marking and sweeping with program execution. The collector uses barriers to preserve a tri‑color invariant: black (already traversed), gray (reachable but not fully traversed) and white (unmarked) objects. The barrier detects when the mutator breaks the invariant and performs a corrective action
lua.org
. The incremental collector uses three tunable parameters
lua.org
:

    GC pause – controls when a new cycle begins. A value of n causes a new cycle when total memory grows by n% over the total after the previous cycle; larger values delay collections
    lua.org
    .

    Step size – approximate number of bytes allocated between incremental steps
    lua.org
    .

    Step multiplier – controls the amount of work per incremental step. n means the collector executes n% units of marking/sweeping per word allocated
    lua.org
    .

Generational GC

Lua 5.2 introduced a generational mode to exploit the empirical “most objects die young” property. Objects are grouped by age into generations. Minor collections visit only young objects, while major collections (less frequent) visit all objects
lua.org
. Lua 5.4 refined the generational algorithm; objects wait two cycles before becoming old. The collector now distinguishes seven states: new, survival, old‑0, old‑1, old, touched‑1 and touched‑2
lua.org
. Forward and back barriers ensure that old objects do not hold references to young objects; if such references appear, the collector either promotes the referenced young object (forward barrier) or marks the old object for scanning (back barrier)
lua.org
. A minor collection promotes survivors (new objects that survive one cycle) to the survival state; after another cycle they become old‑1 and eventually old
lua.org
. In Lua 5.4, major collections are stop‑the‑world; Lua 5.5 aims to make major collections incremental
lua.org
.

Generational GC uses three multipliers
lua.org
:
Parameter	Meaning	Effect
Minor multiplier	When the number of bytes increases by x% over the total after the last major collection, a minor collection runs
lua.org
	Controls frequency of minor collections
Minor‑major multiplier	When the number of bytes from old objects increases by x% over the total after the previous major collection, the collector switches to major collection
lua.org
	Determines when to start a major collection
Major‑minor multiplier	After a major collection, the collector returns to minor collections if it reclaims at least x% of recently allocated bytes
lua.org
	Determines when to switch back to minor collections
Finalizers and Weak Tables

Lua allows finalizers (garbage‑collection metamethods) on tables and userdata. When an object with a __gc metamethod becomes dead, the collector puts it into a list and, at the end of the collection cycle, calls the finalizer with the object as argument
lua.org
. Finalizers run in reverse order of registration
lua.org
and cannot yield or trigger another collection; any errors are converted to warnings
lua.org
. Objects with finalizers are resurrected for at least one additional cycle to allow finalizers to access them
lua.org
.

Lua provides weak tables where keys, values or both are weak references. In a weak‑values table the collector traverses only keys, marking values for removal if their only references are weak
lua.org
. Weak‑keys (ephemeron) tables treat key–value pairs so that a value is reachable only if its key is strongly reachable
lua.org
. All‑weak tables allow collection of both keys and values. Resurrected objects have special rules: they are removed from weak values before finalizers run, and from weak keys only in the next cycle
lua.org
.
Heap vs. Stack Allocation in Lua

Lua uses separate memory spaces for the VM stack and the heap:

    Stack allocation holds local variables and function call frames. When a function is invoked, stack space is automatically allocated for its variables and is freed when the function returns. Stack allocation is contiguous, fast and has limited capacity; exceeding the call stack’s capacity causes overflow
    geeksforgeeks.org
    .

    Heap allocation is used for dynamically created objects such as tables, strings, closures, userdata, coroutines and internal structures. The garbage collector manages heap memory automatically; objects remain in the heap until the GC frees them. Heap allocation is more flexible but slower than stack allocation
    geeksforgeeks.org
    . Lua ensures that all objects subject to garbage collection reside on the heap, while the VM stack contains references to them.

Lualike’s Garbage Collector

The lualike project implements a simplified generational GC for its Dart‑based interpreter. In generational_gc.dart the collector divides objects into a young generation and an old generation and performs minor and major collections
github.com
. All GC‑managed objects implement the GCObject interface with marked and isOld flags
github.com
. Key features include:

    Generational structure – two generations. Objects start in the young generation and are promoted to the old generation after surviving one minor collection
    github.com
    .

    Minor collection – marks reachable objects in the young generation, taking the entire old generation as additional roots to approximate the absence of write barriers
    github.com
    . Survivors are promoted to the old generation.

    Major collection – marks all objects in both generations, converges ephemerons and clears weak tables, separates survivors from dead objects, handles resurrection and calls finalizers
    github.com
    . Major collection is stop‑the‑world and not incremental.

    Memory estimation – uses object counts and heuristics for approximate memory footprints (e.g., each Value contributes 128 bytes plus per‑entry overhead)
    github.com
    rather than the actual byte count that Lua uses.

    Weak table handling – includes simplified support for weak values, weak keys (ephemerons) and all‑weak tables with functions _clearWeakValues, _convergeEphemerons, _clearWeakKeys and _clearAllWeak
    github.com
    .

    Finalization – collects objects with a __gc metamethod into a toBeFinalized set; finalizers are run asynchronously using futures
    github.com
    . The code remarks that finalizers are called in an unspecified order
    github.com
    .

Deviations from Lua’s Specification and Proposed Changes

The current implementation in lualike approximates Lua’s GC but deviates from the official specification in several areas:

    Write barriers and object aging – lualike does not implement forward/back barriers. Instead, minor collections treat all old objects as roots
    github.com
    , defeating the purpose of a generational collector. In Lua, write barriers ensure that references from old objects to young objects are handled without scanning the entire old generation
    lua.org
    . Implementing both forward barriers (promote referenced young object to survival/old‑0 state) and back barriers (mark the old object as “touched”) is essential. In addition, the two‑generation scheme should be replaced with the multi‑phase aging used in Lua 5.4: objects wait two cycles before becoming fully old
    lua.org
    . This requires adding intermediate states (new, survival, old‑0, old‑1, touched‑1, touched‑2) and implementing the state transitions described in the Lua documentation
    lua.org
    .

    Memory accounting – lualike estimates memory usage by counting objects and assigning heuristic sizes
    github.com
    . Lua’s GC uses actual bytes allocated to control collection parameters
    lua.org
    . For better compliance, lualike should track the number of bytes allocated and freed for each object type and base the minor/major thresholds on byte counts rather than object counts. This may require instrumenting object allocation functions to record actual sizes.

    Incremental major collections – lualike performs major collections atomically. Lua’s incremental mode avoids long pauses by interleaving marking and sweeping with program execution and controlling step size and multiplier
    lua.org
    . To more closely follow Lua, lualike could implement an incremental mode: break major collection into small steps, add GC barriers across interpreter operations and expose pause/step multiplier parameters.

    Generational thresholds – the current minor and major multipliers (both default to 100) are not exposed in a way consistent with Lua’s API. In Lua, collectgarbage("setminor"), collectgarbage("setmajor") and collectgarbage("setpause") allow tuning of multipliers. Exposing similar functions (e.g., collectgarbage("setminor", x) in lualike) would improve API compatibility.

    Finalizer semantics – lualike calls finalizers asynchronously via Dart futures and notes that finalizers are called in unspecified order
    github.com
    . Lua specifies that finalizers for objects collected in a cycle are called in reverse order of registration
    lua.org
    and must not yield or run the GC
    lua.org
    . Lualike should:

        Guarantee the reverse‑registration order when iterating _toBeFinalized.

        Avoid using asynchronous tasks for finalization; finalizers should run synchronously at the end of the collection to ensure predictable ordering and to prevent them from overlapping with the mutator. If asynchronous execution is required by the host language, finalizers should at least run to completion before the program continues.

        Prevent finalizers from yielding or triggering nested collections.

    Weak tables and ephemerons – while lualike includes functions to handle weak tables, there are differences:

        Lua’s ephemeron semantics require that a value in a weak‑keys table is kept alive only if its key is reachable outside of any ephemeron table
        lua.org
        . Lualike’s _convergeEphemerons converges ephemerons but uses a fixed maximum iteration count (100), which might miss some convergence cases. It should implement the full fixed‑point algorithm with no arbitrary iteration limit and follow Lua’s semantics for removal of keys after finalization
        lua.org
        .

        When the weakness of a table’s __mode changes, Lua notes that the change may not take effect until the next collection
        lua.org
        . Lualike does not currently support dynamic changes of weakness; adding this capability and ensuring proper timing will improve compliance.

    Resurrection and metamethod registration – lualike resurrects objects with __gc by keeping them alive for one cycle. Lua requires that only objects whose metatable included __gc at the time of setting the metatable are marked for finalization
    lua.org
    . Changing a metatable after object creation should not retroactively mark it. Lualike should enforce this rule. Moreover, Lua collects resurrected objects only when they are dead and not marked for finalization
    lua.org
    ; lualike should ensure that resurrected objects can re‑register for finalization only if their metamethods were set before resurrection.

    Stack vs Heap – lualike must ensure that only GC‑managed objects reside on the heap; objects on the interpreter’s stack should contain references but not own GC‑managed memory. This distinction is crucial because the GC must not scan memory used for local variables or call frames on the host language’s stack; it must treat them as roots instead. Ensuring this separation will improve correctness and performance.

Conclusion

Lua’s garbage collector evolved from a simple mark‑and‑sweep algorithm to a sophisticated incremental generational system. The official specification details how objects are aged, how write barriers maintain invariants, and how finalizers and weak tables are handled

lua.org

Lualike’s current implementation captures the general idea of generational collection but diverges from Lua’s behaviour in several important ways: missing write barriers and multi‑state aging, approximate memory accounting, stop‑the‑world major collections, simplified finalizer ordering, and partial support for weak tables. Performance Characteristics and Limitations

The current GC design is functional but has some performance drawbacks and compliance gaps:

    No write barrier for generational GC: In a true generational collector, the mutator must record pointers from old objects into young objects (“remembered sets”) so that minor collections only scan relevant old-to-young references. The current implementation lacks this optimization – it conservatively treats all old-generation objects as roots during a minor GC
    GitHub
    . This means every minor collection must traverse references in every old object, negating some of the efficiency gains of generational GC. As the heap grows, minor GCs become slower because they effectively perform a full trace of all old objects (many of which may have no references to young objects). The code even acknowledges this: “In a real generational GC, we’d need a write barrier… For now, we’ll just consider all old-gen objects as roots”
    GitHub
    . This is a major source of slowdown in the current design.

    Stop-the-world collection and lack of incremental steps: Both minor and major collections are executed fully on the main thread without preemption. The major GC in particular is stop-the-world by design (it traverses the entire object graph and sweeps in one go) and can introduce noticeable pauses if the heap is large. Although the GC supports an “incremental” mode flag, in practice it does not break the mark-and-sweep work into smaller units – the collectgarbage("step") call simply triggers a minor GC or simulates an allocation, rather than executing a true partial mark step
    GitHub
    . There is a stepSize parameter defined in the GC, but it isn’t yet used to chunk the GC work. As a result, the collector is effectively fully incremental only at the generation level (many minors vs occasional majors), but not incremental within a single mark/sweep phase. This can lead to frame drops or latency if a GC cycle runs long.

    Coarse memory accounting: The use of estimated object sizes (“memory credits”) instead of actual byte tracking can impact when collections are triggered. Because the credit values are heuristic, the GC might run too often or not often enough for certain workloads. For example, every object is given a base overhead (e.g. 64 bytes per GCObject plus 128 bytes for a table Value, etc.)
    GitHub
    GitHub
    . These constants may not perfectly correlate with real memory usage, especially if large Dart objects (like long strings or external resources) are involved. The result is that the threshold calculations (minorMultiplier, majorMultiplier) might be off, potentially causing suboptimal timing of collections. In extreme cases, memory could grow significantly before the GC notices (if actual usage outruns the estimated credits), or the GC might run too eagerly (wasting time on small heaps). This is an inherent limitation without low-level memory access in Dart
    GitHub
    . The team has mentioned possibly using typed data or other tricks to track bytes more directly, but currently it remains an approximation.

    Simplified finalizer and resurrection handling: Lua’s GC requires careful coordination to handle object resurrection (when a finalizer stores a reference to an object that was about to be collected). The current implementation takes a simplified approach: it moves finalizable objects to the survivor list for one extra cycle
    GitHub
    GitHub
    and uses _alreadyFinalized to prevent re-running a finalizer
    GitHub
    . This covers the basics, but it may not capture all corner cases of Lua’s specification. For instance, in Lua the order of finalizer execution is the reverse of object creation order (to handle dependencies), whereas LuaLike currently calls finalizers in an “unspecified” order
    GitHub
    . Also, the GC sets _cycleComplete true only after finalizers finish, which is good, but if a finalizer resurrects an object by storing it in a global, the spec demands that object be treated as a new object (usually moving it to the old gen and not finalizing it again). It’s worth double-checking that the current design matches these expectations. The project notes indicate resurrection is handled in a “simplified” way compared to Lua
    GitHub
    , so this might need review for full compliance.

    Weak table handling completeness: While the implementation for weak keys/values exists, it should be validated against Lua’s behavior in complex scenarios. Ephemeron (weak-key) tables in particular are tricky: the iterative marking logic needs to correctly handle chains of dependencies. The provided _convergeEphemerons() loop tries up to 100 iterations
    GitHub
    GitHub
    , which should usually suffice, but we should ensure this won’t miss cases (Lua’s algorithm runs until convergence with no fixed iteration limit). Also, the current code clears the weakValuesTables, ephemeronTables, etc., at the end of each major cycle
    GitHub
    GitHub
    , which is correct, but it relies on those lists being rebuilt each GC. That means if a table remains weak across collections, it must be re-detected as weak in each new cycle (which the marking phase does). This is fine, just something to keep in mind for performance. Overall, if tests pass, the weak table support is likely correct but any discrepancies with Lua’s weak reference semantics should be identified and fixed for compliance.

    Potential inefficiencies in data structures: The GC currently keeps all managed objects in two Dart List<GCObject> lists (one for each generation). Operations like promoting survivors involve list removals and additions for each object
    GitHub
    . Dart’s list removal is an O(n) operation, so promoting a large number of objects one-by-one could add overhead. Similarly, building new survivor lists and clearing old lists each cycle incurs allocations and copying. These overheads are probably minor compared to the cost of marking, but they could be optimized (for example, by batch-moving survivors to the old generation in one go instead of calling promote() per object). Another minor point is that the GC uses a linear scan to find references (e.g., iterating through map entries and list items in _discover
    GitHub

Needed Updates for Compliance and Performance

To make the garbage collector fully compliant with Lua’s specifications and to improve its performance, several updates are recommended:

    Implement a write barrier (remembered set) for generational GC: This is perhaps the most impactful performance improvement. By recording pointers from old objects to young objects as they occur, the GC can avoid scanning the entire old generation on every minor collection. Concretely, this means modifying the LuaLike runtime so that any time an old-generation object (e.g., an old table or closure environment) has a reference to a new Value stored into it, we log that object in a “remembered set.” Then, a minor GC would only add those recorded old objects (and the usual root set) to its marking roots, instead of all old objects. This update would dramatically cut down the work done in minor GCs, especially as the heap grows. The comment in the code acknowledges this as a “real generational GC” requirement
    GitHub
    . Implementing a write barrier in Dart might involve adding checks in the Value class’s setters or in table assignment operations (like in the RawSetFunction) to call something like GC.remember(oldObj) whenever a young Value is assigned to an old object. With this in place, minor collections remain very fast and scalable, because the cost is proportional to the amount of recently mutated old->young references, not the entire old heap.

    Make the collector more incremental or asynchronous: To reduce GC pause times, the mark-and-sweep work should be breakable into smaller units. There are a couple of approaches:

        Incremental Marking: Implement the GC’s incremental mode properly, using the stepSize and related parameters to perform marking in stages. In Lua’s incremental collector, after a GC cycle is initiated, the interpreter alternates between running a slice of GC work and running normal code, based on an internal “debt” or credit system. We could emulate this by adjusting GenerationalGCManager.collect() to not always complete the entire mark+sweep in one call. Instead, it could maintain a state (e.g., GC phase and which objects have been marked so far) and a budget of work (perhaps measured in number of objects or estimated bytes to mark per step, guided by the stepSize). The simulateAllocation() mechanism is a starting point – it already triggers a minor GC when a certain byte threshold is exceeded
        GitHub
        . We could extend this concept: each call to simulateAllocation(n) could perform a bounded amount of marking (setting a limit on how many objects or table entries to traverse, proportional to n). This way, long collections are split across multiple allocate cycles. Implementing this would require more complex GC state management (tri-color marking, managing a mark stack between steps, etc.), but it would bring LuaLike closer to Lua’s true incremental GC behavior.

        Asynchronous GC using Dart concurrency: Another angle is to move some GC work off the main thread. Dart doesn’t allow threads with shared memory, but we could use an Isolate to perform GC in parallel. This is non-trivial – we’d have to snapshot or transfer the object graph to the isolate, or else design the system such that the entire VM (interpreter + GC) runs in a background isolate. If the interpreter itself runs on a background isolate (which is common in Flutter or server-side use, to avoid blocking the UI isolate), then GC pauses may be less of an issue for the UI. If that’s not the case, we could consider spawning an isolate to handle the mark phase: for example, send it a list of root objects or a representation of pointers, have it compute the reachable set, then return a list of objects to free. However, the overhead of serialization and the complexity of handling mutations during the parallel mark would be significant. A safer compromise might be using asynchronous cooperative multitasking on the same isolate: for instance, use await Future.delayed(Duration.zero) or Dart’s microtask queue to periodically yield control during a long GC. The major collection function is already async and awaits finalizer calls
        GitHub
        , so inserting additional await points (e.g., after marking every N objects) could allow the event loop to process other tasks briefly. We’d need to ensure that the interpreter isn’t actively mutating the object graph in those moments (perhaps pause script execution while GC is mid-cycle). This doesn’t give true parallelism, but it can prevent a single long GC from stalling the event loop completely.

    Improve memory usage accounting (or integrate real byte tracking): To make the GC more predictable and aligned with actual memory, we should refine the memory credit system. This could involve:

        Tuning the constants used in estimateMemoryUse() to better reflect typical object sizes. For example, if profiling shows that certain objects (tables, closures, etc.) use more memory than estimated, we can increase their credit cost so the GC triggers appropriately. The current values (e.g., 48 bytes per table entry, 40 bytes per environment variable, etc.
        GitHub
        GitHub
        ) may need adjustment as the project matures.

        Accounting for external memory or large Dart objects. If Value.raw can hold, say, a large string or an external file handle, those aren’t fully captured by the fixed overhead. One idea is to incorporate the length of strings or lists into the estimate (e.g., if a Value holds a Dart List or String, count its length). Another idea the team floated is using typed data as a form of manual memory tracking. For instance, one could allocate a Uint8List of a certain size to represent an object’s memory footprint; Dart’s own GC would then account for the actual bytes. This is a bit hacky and could bloat memory usage, but it would force the Dart runtime to be aware of how “big” a Lua object is. A simpler route is to periodically call Dart’s ProcessInfo or service APIs to get total heap usage as a sanity check (if running in a VM where that’s available), but that’s more for diagnostics than on-the-fly GC triggers.

        Exposing memory metrics and tuning to the user. Lua’s collectgarbage("count") returns the memory in use (in KB) and some extra info. LuaLike currently returns the estimated KBytes and the current GC step multiplier
        GitHub
        . We should ensure this is documented as an estimate. In the future, if we manage to get more accurate numbers (via native extensions or similar), we could update this. For compliance, the overall behavior (trigger thresholds, etc.) should roughly match Lua’s default GC settings when running equivalent code.

    Finalize weak-reference support and tests: To be fully compliant, we should verify that the weak table behavior matches Lua in all cases. This includes combinations of weak keys and values, metamethod interactions, and edge cases like a weak table being part of an object’s finalizer (Lua’s GC has to handle the scenario where a dying object’s finalizer might insert it into a weak table, etc.). Since the code already implements the core logic for weak keys/values clearing
    GitHub
    GitHub
    , the main task is testing and fine-tuning. Expanding the unit tests (like the existing comprehensive_weak_test.dart
    GitHub
    GitHub
    ) to cover more complex chains of references will increase confidence that our GC is correctly handling ephemerons. If any discrepancy is found (e.g., our GC might clear an entry one cycle too late or early compared to PUC Lua), we’d need to adjust the algorithm (possibly the order of certain phases or how we treat values marked during ephemeron convergence). So far, it appears the implementation aligns with the Lua 5.4 spec – this update is about ensuring complete compliance in tricky cases.

    Strengthen finalizer ordering and resurrection semantics: To align with Lua’s behavior, consider refining how finalizers are processed:

        In Lua, objects to be finalized are moved to a separate list in the order they were found unreachable, and then their finalizers are called after the sweep, in that same order (which effectively means older objects tend to have their finalizers run later). We might want to mimic this more closely. Currently, LuaLike calls finalizers in whatever order the _toBeFinalized set is iterated
        GitHub
        , which in Dart’s set is unspecified (could be insertion order, but not guaranteed). Switching to a list to preserve order of discovery (or explicitly sorting by some age heuristic) could be a minor tweak for compliance.

        If a finalizer resurrects an object (by storing it somewhere reachable), Lua’s rule is that the object is considered “resurrected” and will not be collected in this cycle. Our implementation already does this by adding the object to survivors when marking it for finalization
        GitHub
        . We should double-check that if resurrection happens after the finalizer runs, the object is properly marked alive again. In practice, since we don’t actually free the object until the next cycle, a resurrected object will simply remain in the heap (marked as old generation and with _alreadyFinalized true). We need to ensure the next GC cycle treats it as a normal live object. It likely does, because by then some root will reference it. To fully comply, we should also respect the Lua nuance that an object’s finalizer runs at most once, even if it’s resurrected and dies again – our _alreadyFinalized set enforces that
        GitHub
        . All these points seem to be handled, but a thorough audit and perhaps tests for scenarios of resurrection would be wise.

        One more Lua detail: finalizers are not called for objects that never became fully constructed (in Lua, if an object’s creation throws an error, the GC won’t call its __gc). In LuaLike, if an object with a __gc metamethod is created and immediately abandoned, our GC would still treat it normally. This is probably fine (such scenarios are rare in pure Lua code).

    Optimize GC data structures and algorithms: While not as critical as the above items, there are some low-hanging optimizations:

        Batch promotions: Instead of promoting each survivor one by one during minor GC, we can collect all survivors and then move them to the old generation in bulk. The current code does extra work by calling promote() on each survivor
        GitHub

        Avoid repeated allocations when building root sets or survivor lists. For example, _markGeneration currently takes a list of roots to begin marking
        GitHub
        . In minor GCs, it creates a minorRoots list combining user-provided roots and all oldGen objects
        GitHub
        . If we implement a remembered set, we won’t need to include all old objects, but we will still be constructing some list of roots. We should reuse lists or preallocate buffers if possible to cut down on garbage in the GC implementation itself (ironically, the GC code creating a lot of short-lived Dart objects can cause some overhead).

        Tailor the logging/debugging: In production mode, GC logging should be minimal or off, to avoid the cost of string building for debug messages. The current design already uses a Logger.debug which presumably can be disabled or filtered by category. This isn’t a direct GC logic issue, but it affects performance when running with debug output on. Ensuring that heavy debug logs (like per-entry traversal logs
        GitHub
        ) are off in release builds will help performance.

        Investigate any Dart-specific GC interactions. Because LuaLike holds all GCObjects in Dart collections, the Dart VM’s own garbage collector will treat that as one big strongly-referenced graph. There’s a possibility that a large LuaLike heap could put pressure on Dart’s GC as well (since none of those objects can be freed by Dart until LuaLike’s GC explicitly drops them). This is inherent to implementing a managed runtime on top of another GC. One thing to verify is that when LuaLike frees objects (e.g., removing them from youngGen.objects and oldGen.objects), there are no lingering references that would keep them around. It appears the code correctly removes dead objects from the lists and nulls out any references in structures like upvalues or environment boxes during free() calls
        GitHub
        . As a result, those Dart objects should become unreachable for the Dart VM and get collected on its side. Performance Characteristics and Limitations

The current GC design is functional but has some performance drawbacks and compliance gaps:

    No write barrier for generational GC: In a true generational collector, the mutator must record pointers from old objects into young objects (“remembered sets”) so that minor collections only scan relevant old-to-young references. The current implementation lacks this optimization – it conservatively treats all old-generation objects as roots during a minor GC
    GitHub
    . This means every minor collection must traverse references in every old object, negating some of the efficiency gains of generational GC. As the heap grows, minor GCs become slower because they effectively perform a full trace of all old objects (many of which may have no references to young objects). The code even acknowledges this: “In a real generational GC, we’d need a write barrier… For now, we’ll just consider all old-gen objects as roots”
    GitHub
    . This is a major source of slowdown in the current design.

    Stop-the-world collection and lack of incremental steps: Both minor and major collections are executed fully on the main thread without preemption. The major GC in particular is stop-the-world by design (it traverses the entire object graph and sweeps in one go) and can introduce noticeable pauses if the heap is large. Although the GC supports an “incremental” mode flag, in practice it does not break the mark-and-sweep work into smaller units – the collectgarbage("step") call simply triggers a minor GC or simulates an allocation, rather than executing a true partial mark step
    GitHub
    . There is a stepSize parameter defined in the GC, but it isn’t yet used to chunk the GC work. As a result, the collector is effectively fully incremental only at the generation level (many minors vs occasional majors), but not incremental within a single mark/sweep phase. This can lead to frame drops or latency if a GC cycle runs long.

    Coarse memory accounting: The use of estimated object sizes (“memory credits”) instead of actual byte tracking can impact when collections are triggered. Because the credit values are heuristic, the GC might run too often or not often enough for certain workloads. For example, every object is given a base overhead (e.g. 64 bytes per GCObject plus 128 bytes for a table Value, etc.)
    GitHub
    GitHub
    . These constants may not perfectly correlate with real memory usage, especially if large Dart objects (like long strings or external resources) are involved. The result is that the threshold calculations (minorMultiplier, majorMultiplier) might be off, potentially causing suboptimal timing of collections. In extreme cases, memory could grow significantly before the GC notices (if actual usage outruns the estimated credits), or the GC might run too eagerly (wasting time on small heaps). This is an inherent limitation without low-level memory access in Dart
    GitHub
    . The team has mentioned possibly using typed data or other tricks to track bytes more directly, but currently it remains an approximation.

    Simplified finalizer and resurrection handling: Lua’s GC requires careful coordination to handle object resurrection (when a finalizer stores a reference to an object that was about to be collected). The current implementation takes a simplified approach: it moves finalizable objects to the survivor list for one extra cycle
    GitHub
    GitHub
    and uses _alreadyFinalized to prevent re-running a finalizer
    GitHub
    . This covers the basics, but it may not capture all corner cases of Lua’s specification. For instance, in Lua the order of finalizer execution is the reverse of object creation order (to handle dependencies), whereas LuaLike currently calls finalizers in an “unspecified” order
    GitHub
    . Also, the GC sets _cycleComplete true only after finalizers finish, which is good, but if a finalizer resurrects an object by storing it in a global, the spec demands that object be treated as a new object (usually moving it to the old gen and not finalizing it again). It’s worth double-checking that the current design matches these expectations. The project notes indicate resurrection is handled in a “simplified” way compared to Lua
    GitHub

    Weak table handling completeness: While the implementation for weak keys/values exists, it should be validated against Lua’s behavior in complex scenarios.     GitHub

    GitHub
    GitHub

Needed Updates for Compliance and Performance

To make the garbage collector fully compliant with Lua’s specifications and to improve its performance, several updates are recommended:

    Implement a write barrier (remembered set) for generational GC: This is perhaps the most impactful performance improvement. By recording pointers from old objects to young objects as they occur, the GC can avoid scanning the entire old generation on every minor collection. Concretely, this means modifying the LuaLike runtime so that any time an old-generation object (e.g., an old table or closure environment) has a reference to a new Value stored into it, we log that object in a “remembered set.” Then, a minor GC would only add those recorded old objects (and the usual root set) to its marking roots, instead of all old objects. This update would dramatically cut down the work done in minor GCs, especially as the heap grows. The comment in the code acknowledges this as a “real generational GC” requirement
    GitHub
    . Implementing a write barrier in Dart might involve adding checks in the Value class’s setters or in table assignment operations (like in the RawSetFunction) to call something like GC.remember(oldObj) whenever a young Value is assigned to an old object. With this in place, minor collections remain very fast and scalable, because the cost is proportional to the amount of recently mutated old->young references, not the entire old heap.

    Make the collector more incremental or asynchronous: To reduce GC pause times, the mark-and-sweep work should be breakable into smaller units. There are a couple of approaches:

        Incremental Marking: Implement the GC’s incremental mode properly, using the stepSize and related parameters to perform marking in stages. In Lua’s incremental collector, after a GC cycle is initiated, the interpreter alternates between running a slice of GC work and running normal code, based on an internal “debt” or credit system. We could emulate this by adjusting GenerationalGCManager.collect() to not always complete the entire mark+sweep in one call. Instead, it could maintain a state (e.g., GC phase and which objects have been marked so far) and a budget of work (perhaps measured in number of objects or estimated bytes to mark per step, guided by the stepSize). The simulateAllocation() mechanism is a starting point – it already triggers a minor GC when a certain byte threshold is exceeded
        GitHub
        . We could extend this concept: each call to simulateAllocation(n) could perform a bounded amount of marking (setting a limit on how many objects or table entries to traverse, proportional to n). This way, long collections are split across multiple allocate cycles. Implementing this would require more complex GC state management (tri-color marking, managing a mark stack between steps, etc.), but it would bring LuaLike closer to Lua’s true incremental GC behavior.

        Asynchronous GC using Dart concurrency: Another angle is to move some GC work off the main thread. Dart doesn’t allow threads with shared memory, but we could use an Isolate to perform GC in parallel. This is non-trivial – we’d have to snapshot or transfer the object graph to the isolate, or else design the system such that the entire VM (interpreter + GC) runs in a background isolate. If the interpreter itself runs on a background isolate (which is common in Flutter or server-side use, to avoid blocking the UI isolate), then GC pauses may be less of an issue for the UI. If that’s not the case, we could consider spawning an isolate to handle the mark phase: for example, send it a list of root objects or a representation of pointers, have it compute the reachable set, then return a list of objects to free. However, the overhead of serialization and the complexity of handling mutations during the parallel mark would be significant. A safer compromise might be using asynchronous cooperative multitasking on the same isolate: for instance, use await Future.delayed(Duration.zero) or Dart’s microtask queue to periodically yield control during a long GC. The major collection function is already async and awaits finalizer calls
        GitHub
        , so inserting additional await points (e.g., after marking every N objects) could allow the event loop to process other tasks briefly. We’d need to ensure that the interpreter isn’t actively mutating the object graph in those moments (perhaps pause script execution while GC is mid-cycle). This doesn’t give true parallelism, but it can prevent a single long GC from stalling the event loop completely.

    Improve memory usage accounting (or integrate real byte tracking): To make the GC more predictable and aligned with actual memory, we should refine the memory credit system. This could involve:

        Tuning the constants used in estimateMemoryUse() to better reflect typical object sizes. For example, if profiling shows that certain objects (tables, closures, etc.) use more memory than estimated, we can increase their credit cost so the GC triggers appropriately. The current values (e.g., 48 bytes per table entry, 40 bytes per environment variable, etc.
        GitHub
        GitHub
        ) may need adjustment as the project matures.

        Accounting for external memory or large Dart objects. If Value.raw can hold, say, a large string or an external file handle, those aren’t fully captured by the fixed overhead. One idea is to incorporate the length of strings or lists into the estimate (e.g., if a Value holds a Dart List or String, count its length). Another idea the team floated is using typed data as a form of manual memory tracking. For instance, one could allocate a Uint8List of a certain size to represent an object’s memory footprint; Dart’s own GC would then account for the actual bytes. This is a bit hacky and could bloat memory usage, but it would force the Dart runtime to be aware of how “big” a Lua object is. A simpler route is to periodically call Dart’s ProcessInfo or service APIs to get total heap usage as a sanity check (if running in a VM where that’s available), but that’s more for diagnostics than on-the-fly GC triggers.

        Exposing memory metrics and tuning to the user. Lua’s collectgarbage("count") returns the memory in use (in KB) and some extra info. LuaLike currently returns the estimated KBytes and the current GC step multiplier
        GitHub
        . We should ensure this is documented as an estimate. In the future, if we manage to get more accurate numbers (via native extensions or similar), we could update this. For compliance, the overall behavior (trigger thresholds, etc.) should roughly match Lua’s default GC settings when running equivalent code.

    Finalize weak-reference support and tests: To be fully compliant, we should verify that the weak table behavior matches Lua in all cases. This includes combinations of weak keys and values, metamethod interactions, and edge cases like a weak table being part of an object’s finalizer (Lua’s GC has to handle the scenario where a dying object’s finalizer might insert it into a weak table, etc.). Since the code already implements the core logic for weak keys/values clearing
    GitHub
    GitHub
    , the main task is testing and fine-tuning. Expanding the unit tests (like the existing comprehensive_weak_test.dart
    GitHub
    GitHub
    ) to cover more complex chains of references will increase confidence that our GC is correctly handling ephemerons. If any discrepancy is found (e.g., our GC might clear an entry one cycle too late or early compared to PUC Lua), we’d need to adjust the algorithm (possibly the order of certain phases or how we treat values marked during ephemeron convergence). So far, it appears the implementation aligns with the Lua 5.4 spec – this update is about ensuring complete compliance in tricky cases.

    Strengthen finalizer ordering and resurrection semantics: To align with Lua’s behavior, consider refining how finalizers are processed:

        In Lua, objects to be finalized are moved to a separate list in the order they were found unreachable, and then their finalizers are called after the sweep, in that same order (which effectively means older objects tend to have their finalizers run later). We might want to mimic this more closely. Currently, LuaLike calls finalizers in whatever order the _toBeFinalized set is iterated
        GitHub
        , which in Dart’s set is unspecified (could be insertion order, but not guaranteed). Switching to a list to preserve order of discovery (or explicitly sorting by some age heuristic) could be a minor tweak for compliance.

        If a finalizer resurrects an object (by storing it somewhere reachable), Lua’s rule is that the object is considered “resurrected” and will not be collected in this cycle. Our implementation already does this by adding the object to survivors when marking it for finalization
        GitHub
        . We should double-check that if resurrection happens after the finalizer runs, the object is properly marked alive again. In practice, since we don’t actually free the object until the next cycle, a resurrected object will simply remain in the heap (marked as old generation and with _alreadyFinalized true). We need to ensure the next GC cycle treats it as a normal live object. It likely does, because by then some root will reference it. To fully comply, we should also respect the Lua nuance that an object’s finalizer runs at most once, even if it’s resurrected and dies again – our _alreadyFinalized set enforces that
        GitHub
        . All these points seem to be handled, but a thorough audit and perhaps tests for scenarios of resurrection would be wise.

        One more Lua detail: finalizers are not called for objects that never became fully constructed (in Lua, if an object’s creation throws an error, the GC won’t call its __gc). In LuaLike, if an object with a __gc metamethod is created and immediately abandoned, our GC would still treat it normally. This is probably fine (such scenarios are rare in pure Lua code).

    Optimize GC data structures and algorithms: While not as critical as the above items, there are some low-hanging optimizations:

        Batch promotions: Instead of promoting each survivor one by one during minor GC, we can collect all survivors and then move them to the old generation in bulk. The current code does extra work by calling promote() on each survivor
        GitHub
        , which removes it from the young list one at a time. We could simply clear the young generation list and concatenate survivors onto the old generation list (setting their isOld flag). This would reduce overhead when hundreds of objects survive a minor GC.

        Avoid repeated allocations when building root sets or survivor lists. For example, _markGeneration currently takes a list of roots to begin marking
        GitHub
        . In minor GCs, it creates a minorRoots list combining user-provided roots and all oldGen objects
        GitHub
        . If we implement a remembered set, we won’t need to include all old objects, but we will still be constructing some list of roots. We should reuse lists or preallocate buffers if possible to cut down on garbage in the GC implementation itself (ironically, the GC code creating a lot of short-lived Dart objects can cause some overhead).

        Tailor the logging/debugging: In production mode, GC logging should be minimal or off, to avoid the cost of string building for debug messages. The current design already uses a Logger.debug which presumably can be disabled or filtered by category. This isn’t a direct GC logic issue, but it affects performance when running with debug output on. Ensuring that heavy debug logs (like per-entry traversal logs
        GitHub
        ) are off in release builds will help performance.

        Investigate any Dart-specific GC interactions. Because LuaLike holds all GCObjects in Dart collections, the Dart VM’s own garbage collector will treat that as one big strongly-referenced graph. There’s a possibility that a large LuaLike heap could put pressure on Dart’s GC as well (since none of those objects can be freed by Dart until LuaLike’s GC explicitly drops them). This is inherent to implementing a managed runtime on top of another GC. One thing to verify is that when LuaLike frees objects (e.g., removing them from youngGen.objects and oldGen.objects), there are no lingering references that would keep them around. It appears the code correctly removes dead objects from the lists and nulls out any references in structures like upvalues or environment boxes during free() calls
        GitHub
        . As a result, those Dart objects should become unreachable for the Dart VM and get collected on its side. We should continue to be mindful of retaining references in the implementation (for instance, the _alreadyFinalized and _toBeFinalized sets should remove entries promptly to avoid memory leaks).
