# builder_demo — annotations, builder, and library registration

Demonstrates the four mechanisms for documenting Lua-facing APIs from Dart:

| Mechanism | What it produces | How it's registered |
|-----------|-----------------|---------------------|
| `@TableSchema()` / `@SchemaField()` annotations | `TableDoc` + `FieldDoc` constants (via `build_runner`) | `context.describeTable()` |
| `BuiltinFunction` subclass with `doc` override | `FunctionDoc` with params, return info, examples | `context.define()` |
| `ValueClass` (constructor + metatable) | A Lua-side `newXxx()` constructor | `context.define()` |
| Plain Dart constants | Documented via explicit `FunctionDoc` | `context.define()` + `context.describe()` |

## Project structure

```
lib/
  src/
    plugin_info_schema.dart              # @TableSchema-annotated classes
    plugin_info_schema.table_schema.g.dart  # generated by build_runner
    plugin_api.dart                      # BuiltinFunction subclasses + ValueClass
  plugin_api_library.dart                # Library subclass: registers everything
bin/
  render.dart                            # Entry point: renders annotations + JSON
tool/
  generate.dart                          # One-step: build + render
build.yaml                               # build_runner config
```

## Usage

```sh
# 1. Generate .table_schema.g.dart from annotated classes
dart run build_runner build

# 2. Render LuaLS annotations and JSON manifest
dart run bin/render.dart

# Or do both in one step:
dart run tool/generate.dart
```

## What the annotated classes cover

Three classes in `lib/src/plugin_info_schema.dart` exercise every `@SchemaField` option:

| Class | What it demonstrates |
|-------|---------------------|
| **`SettingsEntry`** | `required`, `defaultValue`, `group`, `dependsOn`, `min`, `max`, `step`, `choices`, and explicit `type` overrides (`"boolean"\|"string"\|"slider"\|"select"`, `number`, `string[]`) |
| **`PluginManifest`** | All inferred types (`string`, `boolean`, `string[]`, `table`), nullable fields, nested type references (`SettingsEntry[]`) |
| **`ColorPalette`** | `@TableSchema(name: 'ColorScheme')` to override the documented type name independently of the Dart class name |

## What the library registers

`PluginApiLibrary` in `lib/plugin_api_library.dart` registers four kinds of exports via `LibraryRegistrationContext`:

- **Constants** — `VERSION` (int), `API_NAME` (string) via `define()`, documented via `describe()`
- **Functions** — `discover()`, `resolveDependencies()`, `formatColor()` as `BuiltinFunction` subclasses with auto-collected `FunctionDoc`
- **Class** — `newConfig()` via `ValueClass` (Lua constructor with `__tostring` / `__index` metamethods)
- **Table schemas** — `SettingsEntry`, `PluginManifest`, `ColorScheme` via `describeTable()`

## Output

The render step produces two formats:

- **LuaLS annotations** — `---@class` / `---@field` stubs for editor completion (Neovim, VS Code with LuaLS)
- **JSON manifest** — structured metadata for custom doc generators or CI pipelines
